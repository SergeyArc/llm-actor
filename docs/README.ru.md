**LLM Actor** — это высокопроизводительный оркестратор для обработки запросов к Large Language Models (LLM) в промышленных масштабах. Вдохновленный **моделью акторов**, он решает проблемы «последней мили» при интеграции LLM в продакшн: управление конкурентностью, обеспечение отказоустойчивости, а также **гарантированный структурированный вывод (structured output)** и **вызов инструментов (Tool Calling)** без лишнего кода.

---

## Почему LLM Actor?

Большинство разработчиков начинают с простых вызовов API. Но в продакшне возникают проблемы:
- **Исчерпание лимитов (Rate Limits)**: Отсутствие глобальной координации использования токенов.
- **Сбои провайдеров**: Одна медленная задержка может повесить всё приложение.
- **Ненадёжный парсинг**: Трудно получить **гарантированный структурированный вывод** из сырого текста.
- **Сложные агентские сценарии**: Ручное управление **Tool Calling** (особенно параллельное) ведет к ошибкам.
- **Отсутствие приоритетов**: Фоновые задачи блокируют важные запросы пользователей в UI.

**LLM Actor** исправляет это. Это не просто обертка, а **отказоустойчивый пул воркеров**, стоящий между вашей бизнес-логикой и провайдерами LLM.

---

## Ключевые возможности

- **Высокопроизводительный пул акторов**: Эффективное управление сотнями конкурентных запросов через выделенных воркеров.
- **Интеллектуальная отказоустойчивость**: 
    - **Circuit Breaker**: Обнаруживает сбои провайдера и мгновенно переходит в режим «быстрого отказа» для защиты вашей инфраструктуры.
    - **Экспоненциальная задержка**: Автоматические повторы при сетевых ошибках (429, 502, 503).
    - **Семантическая валидация**: Типизация ответов через Pydantic с авто-повтором при несоответствии схемы.
- **Встроенный цикл вызова инструментов**: Нативная поддержка сложных агентских сценариев. Выполнение инструментов **параллельно** для минимизации задержек.
- **Глобальная очередь приоритетов**: Назначайте приоритеты задачам. Пользовательские запросы всегда обрабатываются первыми.
- **Мульти-провайдерность и Self-Hosted**: 
    - Нативная поддержка: OpenAI, Anthropic, Sber GigaChat.
    - Поддержка прокси: **vLLM**, **Ollama** и любые OpenAI-совместимые эндпоинты.
- **Глубокая наблюдаемость**: Полная интеграция с **OpenTelemetry**. Отслеживайте путь каждого запроса от очереди до ответа провайдера.

## Установка

```bash
# Установка ядра
pip install llm-actor

## Быстрый старт: маштабирование за 60 секунд

Создайте сервис и начните обрабатывать задачи с автоматическим восстановлением:

```python
from llm_actor import LLMActorService, LLMActorSettings, Priority
from pydantic import BaseModel

# 1. Настройка сервиса
service = LLMActorService.from_openai(
    api_key="sk-...", 
    model="gpt-4o",
    settings=LLMActorSettings(LLM_NUM_ACTORS=10) # 10 параллельных воркеров
)

# 2. Определение схемы ответа
class UserProfile(BaseModel):
    name: str
    skills: list[str]

# 3. Использование через контекстный менеджер (авто-старт и авто-стоп)
async with service:
    # 4. Добавление задачи в очередь с высоким приоритетом
    request = service.request(
        "Extract profile from: Alex is a Senior Python Dev with LLM expertise.",
        response_model=UserProfile,
        priority=Priority.HIGH
    )

    # 5. Получение результата (синхронно или асинхронно)
    result = request.get()
    print(f"Найдено: {result.name}, навыки: {result.skills}")
```

## Спроектировано для надежности

| Механизм | Описание |
|---|---|
| **Супервизор акторов** | Если воркер падает, супервизор автоматически перезапускает его. |
| **Backpressure** | Защищает провайдера от перегрузки, ограничивая активные задачи. |
| **Otel Tracing** | Визуализируйте задержки: «время в очереди» против «времени ответа LLM». |

## Философия дизайна

- **Shared Priority Queue**: Единая очередь обеспечивающая приоритезацию.
- **Изоляция сбоев**: Падение воркера не вешает всю систему.
- **Backpressure**: Защита провайдера от перегрузки.
