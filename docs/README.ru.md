**LLM Actor** — это эффективный Actor Pool для **shared self-hosted inference** (vLLM, Ollama) или прокси-сервисов. Мы решаем задачу «последней мили»: управление конкурентными запросами, их приоритизация и гарантированный структурированный вывод без лишнего boilerplate.

---

## Почему LLM Actor?

В то время как облачные провайдеры имеют свои лимиты, промышленный self-hosted инференс (или общие облачные эндпоинты) сталкивается с другими узкими местами:
- **Перегрузка GPU**: Слишком много конкурентных запросов вешают движок инференса.
- **Захват очереди**: Фоновые пакетные задачи блокируют высокоприоритетные запросы пользователей из UI.
- **Ненадежное структурирование**: Получить **гарантированный структурированный вывод (Structured Output)** от локальных моделей сложно.
- **Отсутствие отказоустойчивости**: Одна медленная задержка не должна вешать весь слой оркестрации.

**LLM Actor** решает это через **управление конкурентностью** (пул акторов) и **глобальный Circuit Breaker**. Мы отдаем приоритет **реактивной устойчивости** перед упреждающим подсчетом токенов, что идеально подходит для частной облачной инфраструктуры.

---

## Ключевые возможности

- **Настраиваемый пул акторов**: Эффективная очередь и диспетчеризация запросов через пул управляемых воркеров.
- **Глобальная отказоустойчивость**: 
    - **Circuit Breaker**: Состояние отказа является общим внутри процесса. Если один воркер фиксирует сбой провайдера, весь пул мгновенно переходит в режим «быстрого отказа», защищая вашу инфраструктуру.
    - **Экспоненциальная задержка**: Автоматические повторы при сетевых ошибках (429, 502, 503).
    - **Семантическая валидация**: Типизация ответов через Pydantic с авто-повтором при несоответствии схемы.
- **Встроенный цикл вызова инструментов**: Нативная поддержка сложных агентских сценариев. Выполнение инструментов **параллельно** для минимизации задержек.
- **Глобальная очередь приоритетов**: Назначайте приоритеты задачам. Пользовательские запросы всегда обрабатываются первыми.
- **Поддержка множества провайдеров**: 
    - Нативные адаптеры: OpenAI, Anthropic, Sber GigaChat.
    - Поддержка прокси: **vLLM**, **Ollama** и любые OpenAI-совместимые эндпоинты.
- **Глубокая наблюдаемость**: Полная интеграция с **OpenTelemetry**. Отслеживайте путь каждого запроса от очереди до ответа провайдера.

## Ограничения

Чтобы сохранить `LLM Actor` легковесным и универсальным, мы пошли на определенные архитектурные компромиссы:

- **In-process state**: Состояние Circuit Breaker и очереди локально для процесса. Разработано для single-node развертываний или вертикального масштабирования.
- **Отсутствие проактивного лимитирования токенов**: Мы не считаем токены *до* отправки. Это идеально подходит для self-hosted инференса (vLLM/Ollama), где лимиты TPM не являются главным узким местом. Для внешних API со строгими TPM-лимитами мы полагаемся на **Реактивную устойчивость** (Backoff + CB).
- **Один провайдер на пул**: Пул привязан к одному экземпляру клиента. Для маршрутизации между разными провайдерами с независимыми «предохранителями» используйте отдельные экземпляры `LLMActorService`.

## Быстрый старт: маштабирование за 60 секунд

Создайте сервис и начните обрабатывать задачи с автоматическим восстановлением:

```python
from llm_actor import LLMActorService, LLMActorSettings, Priority
from pydantic import BaseModel

# 1. Настройка сервиса
service = LLMActorService.from_openai(
    api_key="sk-...", 
    model="gpt-4o",
    settings=LLMActorSettings(LLM_NUM_ACTORS=10) # 10 параллельных воркеров
)

# 2. Определение схемы ответа
class UserProfile(BaseModel):
    name: str
    skills: list[str]

# 3. Использование через контекстный менеджер (авто-старт и авто-стоп)
async with service:
    # 4. Добавление задачи в очередь с высоким приоритетом
    request = service.request(
        "Extract profile from: Alex is a Senior Python Dev with LLM expertise.",
        response_model=UserProfile,
        priority=Priority.HIGH
    )

    # 5. Получение результата (синхронно или асинхронно)
    result = request.get()
    print(f"Найдено: {result.name}, навыки: {result.skills}")
```

## Архитектура: Реактивная устойчивость

`LLM Actor` построен как **In-Process Orchestrator**. Вместо сложного упреждающего управления трафиком мы используем **реактивную цепочку**: `Exponential Backoff` (управляется клиентом) -> `Circuit Breaker` (управляется пулом). 

Этот подход минимизирует внутренние накладные расходы и идеально подходит для self-hosted инференса (как vLLM), где TPM-квоты отсутствуют, а реактивной обработки 429 достаточно. Он предотвращает переполнение памяти GPU, ограничивая число активных соединений (`LLM_NUM_ACTORS`) и гарантируя приоритизацию запросов.

## Философия дизайна

- **Shared Priority Queue**: Единая очередь обеспечивающая приоритезацию.
- **Изоляция сбоев**: Падение воркера не вешает всю систему.
- **Backpressure**: Защита провайдера от перегрузки.
